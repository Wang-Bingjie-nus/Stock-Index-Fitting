### 怎样实现功能

- 每一个执行的 Task 都写成一个单独函数。
- 主函数通过调用不同 Task 完成完整流程，每一步都有明确入参和输出，并将输出传递给下一步。
- 每个 Task 都必须有完整 docstring，说明功能、参数、返回值、数据口径、异常情况和限制。
- 每个 Task 都必须可以单独运行和调试。
- 每个 Task 都必须具有 `debug=False` 参数。
- 当 `debug=True` 时：
    - 在 `logs/<run_id>/` 中记录函数名、调用参数、执行时间、数据行数、关键统计值和异常信息。
    - 将该 Task 的主要 DataFrame 输出为 `debug_outputs/<run_id>/<task_name>.csv`。
    - 在 Notebook 中打印简洁但足够定位问题的调试摘要，不能打印完整大表。
- 所有核心函数应返回结构化对象，优先返回DataFrame或由DataFrame组成的字典，禁止只依赖全局变量。
- 每次完整运行生成唯一 `run_id`，格式建议为 `YYYYMMDD_HHMMSS`。
- 所有日期和行情都必须明确 `as_of_date` 或时间范围，避免误用未来数据。
- 所有数据结果尽量保留以下审计字段：
    - `source`
    - `retrieved_at`
    - `as_of_date`
    - `run_id`
- 禁止静默忽略关键异常。关键数据缺失时必须抛出明确异常，或在校验报告中标记为 `ERROR` 并停止后续正式计算。
- 当迅投接口字段结构与预期不一致时，应先检查实际返回结构，再进行兼容处理；禁止编造不存在的接口返回字段。
- Notebook 必须可以从上到下顺序运行。

#### Task7：使用近似法生成满足交易单位的股票篮子

- 函数名：`build_approximate_portfolio(df_theoretical_portfolio, df_trade_status=None, allow_cash=True, rounding_method="nearest", debug=False)`
- 函数入参：理论组合、交易状态、是否允许现金和取整方式
- 函数输出：
    - `df_target_portfolio[stock_code,stock_name,weight_pct,price,target_amount,theoretical_qty,lot_size,target_qty,target_market_value,target_weight,amount_deviation,weight_deviation,tradeable,reason]`

### 必须回答的业务问题

程序最终输出和说明必须能够回答：

1. 指定期货合约和手数对应多少名义金额？
2. 理论上每只成分股应购买多少金额和数量？
3. 考虑最小购买单位后，实际目标数量是多少？
4. 取整后组合总金额比目标金额高或低多少？
5. 哪些股票造成了最大的金额偏差和权重偏差？
6. 哪些偏差来自取整，哪些来自停牌、涨停或未成交？
7. 模拟实际成交后，股票篮子与指数的跟踪差是多少？
8. 同时持有股票篮子和期货空头后，对冲组合损益是多少？
9. 交易成本、除权除息和基差变化分别造成多少影响？
10. 当前简化方法中，哪些操作偏差未来可以优化？
